# WSN 데모 인수인계

작성 위치: `/home/nsl/tinyos-main/apps`

---

## 0. 한 줄 요약

- 라우터 1대(node1, PppRouter) + 센서노드 3대(node2/3/4, TCPEcho)를 static IPv6 + RPL로 연결.
- **HTTP/curl(포트 80) 이 이제 제대로 동작한다.** 모든 엔드포인트(get-temp/voltage/humidity/sensor 등) 정상.
- **UDP 명령 API(포트 7777)** 도 동작하며 가장 견고하다(백업/비교용).
- 이번에 고친 핵심 2가지:
  1. **노드 영구 정지** — TCP 몇 번 쓰면 노드가 멈춰 재부팅해야 했던 문제(송신 버퍼 풀 고갈). 해결.
  2. **일부 HTTP 명령만 실패** — get-voltage 등 **홀수 길이 응답**이 curl에 안 닿던 문제
     (BLIP TCP의 홀수 길이 페이로드 버그). 응답을 짝수로 패딩해서 해결.
  - (자세한 기술 내용은 아래 5번)

## 3. API — curl (HTTP, 포트 80)

HTTP 헤더 없이 **body에 값만** 보낸다. 그래서 curl은 `--http0.9` 가 필요하다.
주소의 `::4` 부분만 `::2`, `::3` 으로 바꾸면 다른 노드.

> 모든 엔드포인트가 정상 동작한다(홀수 길이 버그 수정 후 get-voltage/get-humidity 각각 6/6 성공 확인).
> **노드 재시작 직후 1~2번째 요청은 실패할 수 있다(RPL 워밍업).** 몇 초 뒤 다시 하면 된다.
> 요청은 몇 초 간격으로 천천히 하는 게 좋다(단일 소켓이라 연타하면 일부 놓칠 수 있음).

```bash
# 온도+습도+전압 한 방에 (변환값 + raw)
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-sensor'
```

```bash
# 온도 (섭씨 변환)
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-temp'

# 전압 (V 변환)
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-voltage'

# 습도 (raw)
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-humidity'

# 온도+습도+전압 한 번에
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-sensor'

# LED 상태 읽기
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-led'

# LED 켜기/끄기 (0~7, 하위 3비트 = LED 3개)
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/set-led/7'
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/set-led/0'

# 채널 / 송신출력 / 가동시간 / 통계
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-channel'
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-tx-power'
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-time'
curl --http0.9 -g --max-time 9 'http://[fd00:23:42:1::4]/get-stats'
```

응답 예 (센서값은 **변환값 + raw** 가 같이 나온다):

```text
node 4, temp 25.65 C (raw 6525), seq 9
node 4, hum 33.93 % (raw 1026), seq 9
node 4, batt 2.49 V (raw 3410), seq 9
node 4, seq 8, temp 25.66 C (raw 6526), hum 33.93 % (raw 1026), batt 2.49 V (raw 3409)
node 4, led 7
node 4, requests 41, ok 40, errors 0, sensor_seq 29, samples 29, udp_sent 29, udp_fail 0, heap 1348
```

변환식 (SHT11 / MSP430, `HttpdP.nc`의 `tempCenti/humCenti/voltCenti`):

```text
temp[C]  = -39.6 + 0.01 * raw
hum[%]   = -2.0468 + 0.0367*raw - 1.5955e-6*raw^2
batt[V]  = raw / 4096 * 3.0
```

### curl이 한 번에 안 될 때 (rc 56 / reset)

BLIP TCP는 단일 소켓이라 가끔 첫 시도가 `Connection reset by peer`(rc 56)로 실패한다.
**그냥 다시 실행하거나**, 아래 "성공할 때까지 재시도" 함수를 쓰면 된다(노드는 죽지 않는다).

```bash
# 터미널에 한 번 붙여넣으면 g 라는 명령이 생김 (성공할 때까지 최대 6번 재시도)
# 재시도 간격은 7초: 노드의 소켓 워치독(최대 6초)보다 길게 둬서 매 시도가 깨끗한 소켓을 만나게 함
g() { for i in 1 2 3 4 5 6; do o=$(curl -s --http0.9 -g --max-time 9 "http://[fd00:23:42:1::4]/$1"); [ -n "$o" ] && { echo "$o"; return; }; sleep 7; done; echo "(실패: 다시 시도)"; }

g get-sensor     # ← 이거 하나면 온도+습도+전압 다 나옴 (제일 추천)
g get-led
g set-led/7
g get-stats
```

> 요청은 **몇 초 간격**으로 천천히 하는 게 성공률이 높다. 연타하면 단일 소켓이라 일부 실패한다.

## 4. API — UDP (포트 7777, 가장 안정적 / 백업)

```bash
printf 'get-temp\n'    | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-voltage\n' | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-sensor\n'  | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-led\n'     | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'set-led/7\n'   | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-channel\n' | nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-tx-power\n'| nc -6 -u -w3 fd00:23:42:1::4 7777
printf 'get-stats\n'   | nc -6 -u -w3 fd00:23:42:1::4 7777
```

HTTP와 명령/응답이 동일하다. **UDP는 거의 100% 동작**한다. HTTP가 잘 되더라도 비교/백업용으로 유용하다.

## 5. 이번에 고친 것 (다음 사람용 기술 메모)

원래 문제: "curl을 몇 번 하면 노드가 완전히 멈춰서(ping·UDP·RPL 전부 죽음) 재플래시해야만 살아난다."

원인과 수정:

1. **송신 버퍼 풀 고갈 (핵심 원인)** — `apps/TCPEcho/Makefile` + `tos/lib/net/blip/IPDispatch.h`
   - `N_FRAGMENTS` 기본 12 → TCPEcho에서만 **20** (`PFLAGS += -DN_FRAGMENTS=20`).
   - `IPDispatch.h`는 `#ifndef N_FRAGMENTS` 로 바꿔서 앱별로 -D 오버라이드 가능하게 함.
     **주의: 헤더에서 직접 20으로 올리면 PppRouter가 RAM 오버플로우로 빌드 실패한다**(공유 헤더라서).
     그래서 TCPEcho Makefile에서만 올림. PppRouter는 기본 12 유지.
   - TCP 한 연결이 패킷(데이터/재전송/FIN)을 몰아 보내 FragPool(12개)을 다 써버리면,
     노드가 RPL 제어 패킷조차 못 보내 DODAG에서 떨어지고 **영구히 멈춤**. UDP는 단일 패킷이라
     이 풀을 고갈시키지 않아서 멀쩡했던 것. 20으로 늘리니 더 이상 영구 정지하지 않음.

2. **수신 콜백 안에서 송신하는 재진입 제거** — `tools/tinyos/c/blip/libtcp/tcplib.c`
   - `tcplib_send()` 안의 `tcplib_output()` 호출을 다시 주석 처리(원래 설계대로 타이머에서 송신).
   - 직전 작업자가 이걸 풀어놔서, 수신 처리 도중에 IP/라디오 스택을 재진입 → 간헐적 정지.

3. **ip_malloc 0길이 셀 방어** — `tools/tinyos/c/blip/lib6lowpan/ip_malloc.c`
   - free 블록을 쪼갤 때 남는 조각이 너무 작으면(0이면) 힙 순회가 무한루프에 빠짐.
   - 남는 조각이 boundary tag보다 작으면 쪼개지 않고 통째로 할당하도록 방어 코드 추가.

4. **HTTP 핸들러 / 소켓 정리** — `apps/TCPEcho/HttpdP.nc`
   - HTTP 응답을 status line 없이 **body만** 전송(모트 RAM이 작아서). `--http0.9`로 읽음.
   - 온도(C)·습도(%)·전압(V)을 **변환값 + raw 둘 다** 출력. 변환식은 위 3번 참고.
     (`putCenti`, `tempCenti/humCenti/voltCenti`, `fmtTemp/fmtHum/fmtVolt/fmtSensor` 함수)
   - `get-stats`에 `heap`(ip_malloc 여유) 추가 — 누수 진단용.
   - 소켓 정리: 응답이 ACK되면 빠르게 close, 응답이 안 가면 재전송할 시간을 준 뒤 정리,
     GET이 안 들어오면(유실) accept 워치독이 소켓을 회수 → 단일 소켓이 오래 묶이지 않게 함.
   - **센서 워치독**: SHT11 read의 `readDone`가 유실되면 `sensor_state`가 안 풀려 샘플링이
     영원히 멈추고 `seq`가 고정된다. `SensorTimer.fired()`에서 매 틱마다 멈춘 read를 버리고
     새 샘플을 시작하도록 해서 seq가 계속 증가하게 고침.

5. **응답 즉시전송 금지 (재진입)** — `tools/tinyos/c/blip/libtcp/tcplib.c`
   - `tcplib_send()`에서 `tcplib_output()`을 호출하면 안 된다(주석 유지). 수신 콜백 안에서
     라디오로 송신하는 재진입이 생겨 노드가 간헐적으로 멈춘다. 송신은 타이머 틱에 맡긴다.
     (즉시전송 + N_FRAGMENTS=20 조합도 테스트했으나 노드가 다시 죽었음 → 즉시전송은 쓰지 말 것)

6. **홀수 길이 HTTP 응답이 안 닿던 문제 (★ get-voltage 등 일부만 실패한 진짜 원인)** — `apps/TCPEcho/HttpdP.nc`
   - 증상: 같은 코드 경로인데 `get-temp`(40B, 짝수)는 되고 `get-voltage`(39B)/`get-humidity`(39B,
     홀수)는 curl에서 rc 56으로 100% 실패. UDP로는 셋 다 정상.
   - 원인: BLIP TCP **송신 경로가 홀수 길이 페이로드를 깨뜨린다.** (UDP는 영향 없음.
     `in_cksum.c`에도 "odd length면 깨진다"는 주석이 있을 만큼 홀수 길이 취급이 부실하다.)
     응답 길이의 홀짝이 성공/실패를 갈랐다(빌드에 따라 어느 쪽이 깨지는지 뒤집히기도 함).
   - 해결: `sendResponse()`에서 **응답 길이가 홀수면 `\n` 하나를 덧붙여 짝수로 패딩**.
     이후 get-voltage/get-humidity 각각 0/6 → **6/6** 으로 정상화.

수정 파일 목록:

```text
apps/TCPEcho/HttpdP.nc
apps/TCPEcho/Makefile                    (-DN_FRAGMENTS=20)
tos/lib/net/blip/IPDispatch.h            (N_FRAGMENTS 를 #ifndef 로, 기본 12)
tools/tinyos/c/blip/libtcp/tcplib.c      (tcplib_send 재진입 제거)
tools/tinyos/c/blip/lib6lowpan/ip_malloc.c (split 방어)
```

## 6. 남은 한계 / 알아둘 점

- **HTTP/curl은 이제 모든 엔드포인트가 동작한다**(홀수 길이 버그 수정 후 get-voltage/get-humidity 6/6 확인).
  - 다만 단일 TCP 소켓 + 무선 손실 특성상 **드물게** 한 요청이 rc 56/28로 실패할 수 있으니
    안 되면 몇 초 뒤 다시 하면 된다(3번의 `g` 재시도 함수). 노드는 영구히 죽지 않는다.
- 노드 재부팅(재플래시) 후에는 항상 **60~70초** 기다렸다 테스트. 직후 1~2번째 요청은 워밍업으로 실패 가능.
- TCP를 더 견고하게 하려면 BLIP의 다중 TCP 소켓 / 홀수 길이 송신 / close-reuse 구조를 더 손볼 수 있다(범위 큼).
- 같은 7777 포트를 UDP 명령 응답과 10초 주기 report가 같이 쓴다. PC에서 `nc -6 -u -l 7777`로
  listener를 켜둔 채 명령을 보내면 출력이 섞일 수 있다.

## 7. 문제 생겼을 때

```bash
# 노드가 응답 없으면: 살아있는지부터 확인
ping6 -c1 fd00:23:42:1::4
printf 'get-stats\n' | nc -6 -u -w3 fd00:23:42:1::4 7777   # UDP가 되면 노드는 살아있음

# PPP가 죽었으면 다시 시작
sudo ./start-wsn-static.sh /dev/ttyUSB0 ; sleep 70

# 그래도 안 되면 해당 노드만 다시 굽기 (예: node4)
make -C TCPEcho telosb
sudo make -C TCPEcho telosb reinstall,4 bsl,/dev/ttyUSB3 ; sleep 70
```
